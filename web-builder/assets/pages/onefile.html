Một file CSS — và đọc repo cho rẻ token

Phần lý-do đằng sau quy trình: vì sao thư viện gói trong một file, vì sao không tách nhỏ, và cách AI nên đọc file lớn để mỗi lần thêm/sửa không đốt token vô ích.

📦
Một file là lời hứa sản phẩm — không phải lười gom. web-builder.css tự quảng cáo “self-contained · no build · drop-in một file”. Tách nó ra nhiều file sẽ buộc app dùng phải @import nhiều lần (thêm request, render-blocking) hoặc dựng một bước build — phá đúng thứ skill đang bán. Thứ tách được là demo docs (pages/*.html) vì chúng không ship; còn CSS thì ở nguyên một file.

Vì sao không tách CSS theo từng component

Rule của một component không nằm gọn một khối — nó rải khắp file: token trong :root · khối chính ở giữa · .dark/:where() override chỗ khác · rồi component khác tái dùng nó (hoá đơn tái dùng card). Đo khoảng cách dòng đầu ↔ dòng cuối của mỗi class:

receipt · gọn ~145 dòng colorpicker · ~64 input · rải ~2340 card · rải ~2319 chart · rải ~1463

input/card rải gần cả file — tách vật lý sẽ phải nhân đôi cascade dùng chung hoặc dựng chuỗi @import phụ thuộc thứ tự mong manh. Một file là biểu diễn trung thực của một cascade gắn chặt, không phải nợ kỹ thuật.

AI không đọc lại cả repo mỗi lần

Mỗi phiên chỉ tự nạp CLAUDE.md + bộ nhớ (nhỏ); còn lại đọc theo nhu cầu. Kẻ đốt token thật là 3 file lớn khi lỡ đọc full — nên đừng bảo AI “đọc repo cho hiểu”:

web-builder.css

~2900 dòng ≈ ~40k token nếu đọc full.

app.js · catalog

~16k token mỗi file.

Đọc “ngây thơ” cả ba

~70k token trước khi viết một dòng.

Đọc theo cụm — để grep làm bản đồ

Không Read full web-builder.css. Thay vào đó:

  1. 1

    grep -n 'wb-<comp>'

    Ra toàn bộ dòng của component — kể cả token, dark override và chỗ tái dùng rải rác.

  2. 2

    Gom cụm

    Nhóm các dòng liền nhau thành vài cụm (khối chính · token · override).

  3. 3

    Read từng cụm

    Bằng offset/limit (+vài dòng đệm) — không đọc cửa sổ cố định ±N dòng.

.claude/tools/wb.sh locate <class> làm hộ cả 3 bước trên — in sẵn lệnh Read theo cụm + bản đồ blast-radius (xem Tooling §3). grep tìm mọi lần xuất hiện nên bất chấp component dài hay rải rác; cửa sổ cố định ±N dòng thì sẽ sót rule. receipt đọc ~2k token thay vì 40k (thắng ~20×); input rải rác đọc ~3 cụm ~300 dòng (vẫn thắng ~8×). Việc đọc rộng thì đẩy cho subagent — 40k đó nằm trong context riêng của nó, chỉ bản tóm tắt trở về luồng chính.

🛟
Lỡ đọc sót một lát thì sao? Không ship lỗi âm thầm: validate-sync.sh lúc commit và bước verify sáng/tối làm demo page vỡ mắt thường nếu thiếu token hay dark override — worst case là đọc lại một lượt, không phải bug lọt ra.
