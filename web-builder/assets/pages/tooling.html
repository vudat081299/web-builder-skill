Tooling: serve · verify · hook

Docs của repo này là siêu tập con người đọc được của mọi thứ trong repo: skill + tool + code + tài liệu (README, CLAUDE.md, references…). Trang này lo phần tool — thứ vận hành và bảo vệ skill: server preview serve.py, bộ kiểm validate-sync.sh, hai hook tự chạy, và quy trình /wb-change nối tất cả. Tất cả nằm ở .claude/ và web-builder/assets/.

🛡️
Tooling KHÔNG ship. File duy nhất đi vào app của bạn là web-builder/assets/web-builder.css. serve.py, các hook và validate-sync.sh chỉ là đồ nghề: để dựng, review, dogfood và giữ skill không lệch. Đo mọi thay đổi bằng một câu: skill có tốt lên và còn nhất quán không?

1 · serve.py — dev server no-store

Docs là một app client nhỏ (router fetch các pages/*.html) nên phải chạy qua HTTP — mở index.html bằng file:// sẽ không nạp được trang. serve.py là lớp mỏng bọc http.server, thêm header Cache-Control: no-store, nên reload thường là thấy sửa (không cần hard-refresh); router cũng fetch trang với cache: "no-store".

cd web-builder/assets && python3 serve.py        # → http://127.0.0.1:8777
cd web-builder/assets && python3 serve.py 9000   # truyền cổng để đổi 8777

# Deep-link thẳng tới một trang bằng hash:
#   http://127.0.0.1:8777/#/tables   ·   #/receipt   ·   #/principles
info
Docs nạp Google Fonts (Material Symbols + bộ chọn font) nên cần internet. Chạy python3 -m http.server 8777 cũng được, nhưng nó cache → phải hard-refresh (Cmd/Ctrl+Shift+R) sau mỗi lần sửa. Vị trí: web-builder/assets/serve.py.

2 · validate-sync.sh — bộ tiêu chuẩn bằng shell

Đây là "standards review" viết bằng shell: không cần model nên gần như miễn phí token và gác được mọi commit. Vì repo ship skill (không phải docs site), nó validate cả hai nửa: docs site và skill deliverable. Kết quả: exit 0 = mọi invariant giữ (note nhắc nhở vẫn có thể in ra); exit 2 = một invariant cứng vỡ → in một dòng BLOCK · kèm lý do để sửa tại chỗ (không xoá làm lại).

bash .claude/hooks/validate-sync.sh    # exit 0 = OK · exit 2 + "BLOCK ·" = có chỗ lệch, sửa đi

3 · Hai hook — gác tự động

Hook được wire trong .claude/settings.json, chạy trên máy bạn nên không phải nhớ. Một hook nhắc lúc sửa, một hook chặn lúc commit.

PostToolUse — nhắc cascade

.claude/hooks/wb-sync-reminder.sh · matcher Edit|Write. Vừa chạm web-builder.css là hook chèn checklist 6 nơi vào context để không quên đồng bộ. Hook không tự viết docs (đó là việc của model) — nó chỉ bảo đảm cascade không bị bỏ sót.

PreToolUse — cổng commit

.claude/hooks/pre-commit-gate.sh · matcher Bash. Chỉ gác git commit/git push: chạy validate-sync.sh trước, exit 2 thì chặn — drift không lọt ra. Mọi lệnh Bash khác đi thẳng qua.

Đoạn wiring thật trong .claude/settings.json (matcher + timeout):

{
  "hooks": {
    "PostToolUse": [
      { "matcher": "Edit|Write",
        "hooks": [{ "type": "command",
          "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/wb-sync-reminder.sh", "timeout": 10 }] }
    ],
    "PreToolUse": [
      { "matcher": "Bash",
        "hooks": [{ "type": "command",
          "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/pre-commit-gate.sh", "timeout": 30 }] }
    ]
  }
}

4 · /wb-change — sợi chỉ nối tất cả

Không phải gọi skill bằng tay. Khi bạn tả một thay đổi component bằng lời thường, /wb-change tự trigger (khớp mô tả: "thêm component", "đổi token", "restructure", "sửa wb-*"…) → 6-place sync + guardrails theo sau. CLAUDE.md nạp mỗi turn nên AI luôn biết luật; hai hook trên chạy tự động. Toàn bộ 10 bước / 4 giai đoạn của workflow nằm ở trang Tổng quan (mục "Workflow — pipeline /wb-change") và ở file skill .claude/skills/wb-change/SKILL.md.

bolt
Chỉ "dùng thư viện để dựng UI" không trigger /wb-change — đó là skill web-builder riêng. /wb-change chỉ dành cho việc đổi chính thư viện.
